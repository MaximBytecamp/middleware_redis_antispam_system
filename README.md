# Техническое задание: API «Антиспам для формы заявок»

## 1. Идея проекта

Разработать API-сервис для отправки заявок на консультацию.

Пользователь может отправить заявку, проверить её статус и получить информацию о работе сервиса.

Главная задача проекта — реализовать middleware, которое защищает API от спама и собирает технические метрики с помощью Redis.

---

## 2. Технологии

- Python;
- FastAPI;
- Redis;
- `redis.asyncio`;
- Pydantic;
- Docker Compose;
- стандартный модуль `logging`.

База данных не обязательна. Заявки можно хранить в памяти или в Redis.

# 3. Основные маршруты

## Создание заявки

```http
POST /applications
```

Тело запроса:

```json
{
  "name": "Максим",
  "email": "max@example.com",
  "message": "Хочу записаться на консультацию"
}
```

Ответ:

```json
{
  "id": "app_124",
  "status": "accepted"
}
```


# 4. Middleware №1. Request ID

Для каждого входящего запроса создать уникальный идентификатор.

```text
X-Request-ID: a41750b4-57c1-4d20-a076-b55af00d6238
```

Логика:

- если клиент передал `X-Request-ID`, использовать его;
- иначе сгенерировать UUID;
- сохранить значение в `request.state.request_id`;
- добавить идентификатор в заголовок ответа.

Пример лога:

```text
[a41750b4] POST /applications status=201 time=35.7ms
```

---

# 5. Middleware №2. Rate Limiting через Redis

Ограничить количество запросов от одного IP-адреса.

Правило:

```text
не более 10 запросов за 60 секунд
```

Для каждого клиента хранить в Redis ключ:

```text
rate_limit:192.168.1.25
```

Пример значения:

```text
7
```

При первом запросе необходимо:

```python
INCR rate_limit:192.168.1.25
EXPIRE rate_limit:192.168.1.25 60
```

Если количество запросов превышает лимит, вернуть:

```http
429 Too Many Requests
```

Ответ:

```json
{
  "detail": "Слишком много запросов. Попробуйте позже"
}
```

Дополнительно возвращать заголовки:

```http
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 3
Retry-After: 42
```

Оставшееся время получать через Redis-команду:

```text
TTL rate_limit:192.168.1.25
```

---

# 6. Middleware №3. Защита от повторной отправки заявки

Пользователь не должен отправлять одинаковую заявку несколько раз подряд.

Для этого сформировать хеш из:

- IP-адреса;
- email;
- текста сообщения.

Пример Redis-ключа:

```text
duplicate:192.168.1.25:54ad1f8c
```

После успешной отправки заявки установить ключ:

```text
SET duplicate:192.168.1.25:54ad1f8c 1 EX 120
```

Если такой ключ уже существует, вернуть:

```http
409 Conflict
```

Ответ:

```json
{
  "detail": "Такая заявка уже была отправлена недавно"
}
```

Проверка должна выполняться через middleware.

---

# 7. Middleware №4. Временная блокировка подозрительных клиентов

Если клиент получил ответ `429` три раза в течение пяти минут, временно заблокировать его.

Redis-ключ для подсчёта нарушений:

```text
violations:192.168.1.25
```

Redis-ключ блокировки:

```text
blocked:192.168.1.25
```

Блокировка должна действовать:

```text
10 минут
```

При обращении заблокированного клиента вернуть:

```http
403 Forbidden
```

Ответ:

```json
{
  "detail": "Ваш IP временно заблокирован"
}
```

Дополнительно вернуть:

```http
Retry-After: 540
```

Оставшееся время блокировки получать через `TTL`.

---

# 8. Middleware №5. Сбор статистики в Redis

Каждый запрос должен увеличивать общий счётчик:

```text
stats:requests:total
```

Счётчики по HTTP-методам:

```text
stats:requests:GET
stats:requests:POST
stats:requests:DELETE
```

Количество заблокированных запросов:

```text
stats:requests:blocked
```

Количество ошибок:

```text
stats:responses:4xx
stats:responses:5xx
```

Пример команд:

```text
INCR stats:requests:total
INCR stats:requests:POST
```

Маршрут `/stats` должен получать эти значения из Redis.

---

# 9. Middleware №6. Измерение времени запроса

Для каждого запроса измерить длительность выполнения.

Добавить заголовок:

```http
X-Process-Time: 24.8ms
```

Также сохранять суммарное время всех запросов:

```text
stats:process_time:total
```

И количество измеренных запросов:

```text
stats:process_time:count
```

Среднее время ответа:

```python
average_time = total_time / request_count
```

Результат выводить в `/stats`.

---

# 10. Middleware №7. Режим технических работ через Redis

Состояние технических работ хранить в Redis:

```text
service:maintenance
```

Возможные значения:

```text
0
1
```

Если значение равно `1`, middleware блокирует все маршруты, кроме:

```text
/health
/docs
/openapi.json
```

Ответ:

```http
503 Service Unavailable
```

```json
{
  "detail": "Сервис временно находится на техническом обслуживании"
}
```

Для управления режимом создать маршруты:

```http
POST /admin/maintenance/on
POST /admin/maintenance/off
```

Они должны быть защищены заголовком:

```http
X-Admin-Key: secret123
```

---

# 11. Кэширование через middleware

Для маршрута:

```http
GET /stats
```

реализовать кэширование ответа на 15 секунд.

Redis-ключ:

```text
cache:stats
```

Логика:

1. Middleware проверяет наличие ключа.
2. Если ключ существует, возвращает сохранённый JSON.
3. Если ключа нет, выполняет маршрут.
4. Сохраняет результат в Redis на 15 секунд.

Добавлять заголовок:

```http
X-Cache: HIT
```

или:

```http
X-Cache: MISS
```

# 12. Логирование

Каждый запрос должен выводиться в консоль.

Пример:

```text
[4f23c8a1] POST /applications
IP=192.168.1.25
status=201
time=32.5ms
rate_limit=4/10
cache=MISS
```

Для заблокированного запроса:

```text
[13c7d921] POST /applications
IP=192.168.1.25
status=429
reason=rate_limit_exceeded
```

# 13. Docker Compose

Необходимо запускать приложение и Redis через Docker Compose.

Пример сервисов:

```yaml
services:
  api:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - redis

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```

Приложение должно подключаться к Redis по адресу:

```text
redis://redis:6379
```

# 14. Обработка ошибок

Необходимо предусмотреть:

- Redis недоступен — `503`;
- превышен лимит запросов — `429`;
- клиент временно заблокирован — `403`;
- повторная заявка — `409`;
- заявка не найдена — `404`;
- ошибка валидации — `422`.
